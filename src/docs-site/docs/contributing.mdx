## 1. How to Contribute

<div class="csnCard">

This project welcomes improvements in backend, frontend, AI, extension,
documentation, and DevOps. Clear structure and safe collaboration are
important.

</div>

---

<div class="csnSteps">

<div class="csnStep">

<div class="csnStepTop">
  <div class="csnStepNum">1</div>
  <div class="csnStepTag">Fork</div>
</div>

<h3 class="csnStepTitle">Fork the Repository</h3>

<p class="csnStepDesc">

- Click **Fork** on GitHub
- Clone your fork locally

</p>

</div>

<div class="csnStep">

<div class="csnStepTop">
  <div class="csnStepNum">2</div>
  <div class="csnStepTag">Branch</div>
</div>

<h3 class="csnStepTitle">Create a Branch</h3>

<p class="csnStepDesc">

Use clear branch naming:

- `feature/<topic>`
- `fix/<topic>`
- `docs/<topic>`
- `refactor/<topic>`
- `chore/<topic>`

Example: `feature/url-whitelist-ui`

</p>

</div>

<div class="csnStep">

<div class="csnStepTop">
  <div class="csnStepNum">3</div>
  <div class="csnStepTag">Commit</div>
</div>

<h3 class="csnStepTitle">Follow Conventional Commits</h3>

<p class="csnStepDesc">

Use standard commit prefixes:

- `feat:` new feature
- `fix:` bug fix
- `docs:` documentation change
- `refactor:` code restructuring
- `chore:` maintenance task
- `perf:` performance improvement
- `test:` add/update tests

Example:

`feat: add dataset approval endpoint`  
`fix: correct scan threshold logic`

</p>

</div>

<div class="csnStep">

<div class="csnStepTop">
  <div class="csnStepNum">4</div>
  <div class="csnStepTag">Code</div>
</div>

<h3 class="csnStepTitle">Code Guidelines</h3>

<p class="csnStepDesc">

- Keep functions small and readable
- Use meaningful variable names
- Add comments where logic is non-trivial
- Avoid breaking existing APIs
- Ensure role-based access is respected
- Do not expose sensitive data in logs

</p>

</div>

<div class="csnStep">

<div class="csnStepTop">
  <div class="csnStepNum">5</div>
  <div class="csnStepTag">PR</div>
</div>

<h3 class="csnStepTitle">Pull Request Process</h3>

<p class="csnStepDesc">

When opening a PR:

- Provide a clear description
- Reference related issue (if any)
- Include screenshots (if UI change)
- Mention testing steps
- Ensure build passes
- Ensure no sensitive data committed

PR Checklist:

- Code compiles
- No obvious security issue
- Documentation updated (if needed)
- No debug logs left in production code

</p>

</div>

</div>

## 2. Issue & Labeling Guidelines

<div class="csnCard">

Suggested labels:

- `setup`
- `backend`
- `frontend`
- `extension`
- `ai`
- `docs`
- `security`
- `devops`
- `bug`
- `enhancement`
- `performance`

Use labels consistently to keep backlog organized.

</div>

---

## 3. Security Reporting

<div class="csnCard">

If you discover a security vulnerability:

- Do **NOT** open a public issue
- Contact maintainers privately
- Provide steps to reproduce
- Allow time for patch before disclosure

</div>

---

## 4. AI & Training Contributions

<div class="csnCard">

If contributing to the AI pipeline:

- Do not modify training logic without evaluation
- Provide metric comparison before/after
- Ensure rollback safety
- Keep previous model version intact

</div>

## 6. Contribution Philosophy

<div class="csnCard">

ChildSafeNet prioritizes:

- Stability over rapid change
- Security over convenience
- Explainability over black-box logic
- Clean documentation over hidden behavior

Thank you for helping improve internet safety for children.
